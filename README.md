## 1. Аннотация

Настоящий документ является технической документацией на комплекс программного обеспечения, разработанного для модуля Hulyn WT32-ETH01 V1.4 на базе микроконтроллера ESP32 в рамках проекта РИПАС.

Разработанный комплекс обеспечивает непрерывное высокочастотное пассивное сканирование эфира в диапазоне Bluetooth Low Energy (BLE), формирование структурированных событий сканирования с временны́ми метками, сериализацию данных в формате Protocol Buffers (Protobuf), буферизацию событий в режиме офлайн и передачу данных на сервер-приёмник по протоколу TCP/Ethernet.

Программное обеспечение разработано с использованием фреймворка ESP-IDF версии 5.5.1. Серверная часть реализована на языке Python 3.11. В качестве среды для тестирования сетевого взаимодействия использована виртуальная машина под управлением Ubuntu (Oracle VirtualBox).

Документ содержит описание архитектуры, реализации всех программных модулей, форматов передаваемых данных, инструкции по сборке, прошивке и развёртыванию, а также результаты функционального тестирования.

## 3. Введение

### 3.1. Цели разработки

Разработка направлена на решение следующих задач:

- обеспечить непрерывное высокочастотное пассивное сканирование BLE-эфира с временны́м разрешением не хуже 100 мс;
- реализовать сбор, первичную обработку и буферизацию событий BLE с сохранением в режиме офлайн;
- реализовать синхронизацию системного времени устройства по протоколу SNTP и прикрепление абсолютных UTC-меток ко всем событиям;
- реализовать сериализацию событий в формате Protocol Buffers и передачу групп событий (батчей) по TCP на сервер-приёмник;
- создать серверную часть (Python) для приёма, декодирования и сохранения данных в формате CSV;
- подготовить лабораторный стенд на базе VirtualBox/Ubuntu для тестирования сетевого взаимодействия.

### 3.2. Объект разработки

Объект разработки — программный комплекс, состоящий из:

- прошивки для модуля Hulyn WT32-ETH01 V1.4 (ESP32), разработанной на ESP-IDF v5.5.1;
- серверного приложения на Python 3.11 для приёма и записи данных;
- лабораторного стенда на базе Oracle VirtualBox с гостевой ОС Ubuntu.

### 3.3. Аппаратная платформа

Модуль **Hulyn WT32-ETH01 V1.4**:

- MCU: ESP32 (Xtensa LX6, двухъядерный, 160/240 МГц);
- Ethernet PHY: LAN8720, интерфейс RMII;
- Поддержка: Wi-Fi 802.11 b/g/n, Bluetooth 4.2/BLE;
- Flash: 4 МБ;
- Питание: 5 В (через USB-программатор или внешний блок питания).

**Конфигурация тестового стенда:**
- Ноутбук с ОС Windows 11 (основной хост);
- USB-LAN адаптер (USB → Ethernet) для подключения к модулю;
- Ethernet-кабель: WT32-ETH01 → USB-LAN адаптер → ноутбук;
- USB-программатор для прошивки и мониторинга UART;
- Oracle VirtualBox с гостевой ОС Ubuntu для запуска Python-сервера.

### 3.4. Область применения

Разработанный комплекс предназначен для использования в системах:

- сбора телеметрического радиоокружения (BLE);
- задач позиционирования и построения радиокарт;
- исследования плотности и активности BLE-устройств в заданном пространстве.

## 5. Описание реализации прошивки ESP32

### 5.1. Инициализация Ethernet (`eth.c`)

Ethernet инициализируется функцией `eth_init()` при старте системы. Используется встроенный MAC-контроллер ESP32 (EMAC) с внешним PHY LAN8720.

**Параметры конфигурации:**

| Параметр | Значение | Примечание |
|---|---|---|
| MDC GPIO | 23 | Управляющий сигнал SMI |
| MDIO GPIO | 18 | Данные SMI |
| RMII Clock Mode | `EMAC_CLK_EXT_IN` | Внешний тактовый сигнал от PHY |
| RMII Clock GPIO | 0 | GPIO для приёма тактового сигнала |
| PHY Address | 1 | Адрес LAN8720 на шине SMI |
| PHY Reset GPIO | 16 | GPIO сброса PHY |
| IP-адрес | DHCP | Автоматическое получение от DHCP-сервера хоста |

При получении IP-адреса (событие `IP_EVENT_ETH_GOT_IP`) автоматически запускается синхронизация времени через SNTP.

**Примечание о питании.** При запуске BLE совместно с Ethernet возникает кратковременная просадка напряжения питания, что может вызвать срабатывание аппаратного детектора Brownout. Временное решение — отключение Brownout Detector в `sdkconfig` (`CONFIG_ESP_BROWNOUT_DET=n`). Рекомендуемое постоянное решение — конденсатор 300–400 мкФ между выводами 3V3 и GND, либо отдельный качественный блок питания.

### 5.2. BLE-сканер (`ble.c`)

BLE-сканирование реализовано на стеке **NimBLE**, интегрированном в ESP-IDF. Используется режим **пассивного сканирования** (без отправки SCAN_REQ).

**Параметры сканирования:**

| Параметр | Значение | Пояснение |
|---|---|---|
| Режим | Passive (`params.passive = 1`) | Не отправляет запросы устройствам |
| Интервал сканирования | `0x50` (50 мс, единицы 0.625 мс) | Период повторения окна сканирования |
| Окно сканирования | `0x30` (30 мс) | Длительность активного прослушивания |
| Фильтрация дубликатов | Выключена (`filter_duplicates = 0`) | Все пакеты фиксируются |
| Длительность | `BLE_HS_FOREVER` | Непрерывное сканирование |

При обнаружении устройства вызывается коллбэк `gap_event()` с типом события `BLE_GAP_EVENT_DISC`. Из события извлекаются:

- MAC-адрес устройства (6 байт);
- RSSI в дБм;
- данные рекламного пакета (ADV data, до 31 байта);
- длина ADV-данных.

К событию добавляются временны́е метки из модулей `monotonic` и `time_sync`, а также флаг синхронизации времени. Сформированное событие помещается в очередь FreeRTOS.

При завершении цикла сканирования (`BLE_GAP_EVENT_DISC_COMPLETE`) сканирование автоматически перезапускается.

### 5.3. Синхронизация времени (`time_sync.c`)

Синхронизация времени выполняется по протоколу SNTP с публичным сервером `pool.ntp.org`.

**Алгоритм:**

1. После получения IP-адреса вызывается `time_sync_init()`.
2. ESP-IDF выполняет SNTP-запрос; при успехе вызывается коллбэк `sntp_sync_cb()`.
3. В коллбэке вычисляется UTC-время в миллисекундах: `utc_ms = tv_sec * 1000 + tv_usec / 1000`.
4. Считывается текущее монотонное время: `mono_ms = monotonic_time_ms()`.
5. Вычисляется и сохраняется смещение: `s_time_offset_ms = utc_ms − mono_ms`.
6. Устанавливается флаг `s_time_synced = true`.

**Публичное API модуля:**

```c
void     time_sync_init(void);       // Инициализация SNTP
bool     time_is_synced(void);       // Статус синхронизации
uint64_t time_get_utc_ms(void);      // UTC = mono_ms + offset_ms
int64_t  time_get_offset_ms(void);   // Текущее смещение
```

**Результат:** точность абсолютной временно́й метки UTC составила порядка ±60 мс при нормальной работе сети. Наблюдается медленный нисходящий тренд ~61 мс/ч, связанный с расхождением часов ESP32 и NTP-сервера между сессиями синхронизации (интервал по умолчанию — 1 час).

### 5.4. Формирование батчей (`batch_builder.c`)

Батч — группа BLE-событий, упакованных в одно Protobuf-сообщение `ScanBatch` и отправленных единым TCP-пакетом.

**Параметры:**

| Параметр | Значение | Константа |
|---|---|---|
| Макс. событий в батче | 10 | `MAX_BATCH_EVENTS` |
| Таймаут отправки | 100 мс | `BATCH_TIMEOUT_MS` |
| Макс. размер батча | 640 байт | `BLE_PACKET_MAX_SIZE` |

**Условия отправки (функция `batch_should_send()`):**

- накоплено 10 событий — отправка немедленно;
- прошло 100 мс с момента последней отправки при наличии хотя бы одного события.

**Формат передаваемого сообщения:**

```
[ length_lo ][ length_hi ][ protobuf payload ... ]
  2 байта (little-endian)         N байт
```

Поле `length` содержит длину Protobuf-payload в байтах (без учёта самих 2 байт заголовка). Сервер использует этот заголовок для корректного фрейминга потока TCP.

**Порядковый номер батча** (`batch_seq`) инкрементируется при каждой отправке, что позволяет обнаруживать потери пакетов на стороне сервера.

### 5.5. Кольцевой буфер (`ble_buffer.c`)

При отсутствии TCP-соединения сформированные батчи помещаются в кольцевой буфер в оперативной памяти.

**Параметры:**

| Параметр | Значение | Константа |
|---|---|---|
| Ёмкость буфера | 128 батчей | `BLE_BUFFER_CAPACITY` |
| Макс. размер одного батча | 640 байт | `BLE_PACKET_MAX_SIZE` |
| Суммарный объём ОЗУ | ~82 КБ | 128 × 640 |

**Поведение при переполнении:** при достижении ёмкости 128 записей новые батчи отбрасываются. При восстановлении соединения TCP производится полный сброс буфера (`ble_buffer_flush()`): батчи отправляются в порядке поступления — от наиболее старых к наиболее свежим.

Доступ к буферу защищён мьютексом FreeRTOS (`SemaphoreHandle_t`), что исключает гонки данных между задачами записи (app_task) и чтения (при flush).

### 5.6. TCP-клиент (`tcp_client.c`)

TCP-клиент работает в отдельной задаче FreeRTOS с приоритетом 5.

**Логика работы:**

1. Создаётся TCP-сокет (`AF_INET`, `SOCK_STREAM`).
2. Выполняется подключение к серверу по заданному IP и порту.
3. При успехе: вызывается `transport_attach(sock)`, в очередь отправляется событие `APP_EVENT_TCP_CONNECTED`.
4. Задача периодически (каждые 1 с) проверяет состояние соединения через `transport_is_connected()`.
5. При разрыве: вызывается `transport_detach()`, в очередь отправляется `APP_EVENT_TCP_DISCONNECTED`, выполняется пауза 2 с, затем повтор подключения.

**Настройки подключения** (задаются в `tcp_client.c`):

| Параметр | Значение для Linux/Ubuntu |
|---|---|
| `SERVER_IP` | `"192.168.137.1"` (Windows) / `"192.168.1.2"` (Linux) |
| `SERVER_PORT` | `9000` |

### 5.7. Транспортный уровень (`transport_tcp.c`)

Транспортный уровень абстрагирует операции отправки данных от конкретного сокета.

```c
void transport_attach(int sock);                        // Привязать сокет
void transport_detach(void);                            // Отвязать сокет
bool transport_is_connected(void);                      // Проверить соединение
int  transport_send(const uint8_t *data, size_t len);  // Отправить данные
```

При ошибке отправки (`send()` возвращает отрицательное значение) сокет помечается как недействительный (`s_transport_sock = -1`), что автоматически переводит систему в режим накопления.

## 6. Форматы данных и Protobuf-схемы

### 6.1. Схема Protobuf (`scan.proto`)

```protobuf
syntax = "proto3";
package ripas;

message BleEvent {
  bytes  mac          = 1;  // MAC-адрес устройства, 6 байт
  sint32 rssi         = 2;  // Уровень сигнала, дБм (sint32 для эффективного кодирования)
  bytes  adv_data     = 3;  // Данные рекламного пакета, до 31 байта
  uint64 utc_ms       = 4;  // Абсолютное время UTC, Unix time в мс
  uint64 monotonic_ms = 5;  // Монотонное время от запуска, мс
  bool   time_sync    = 6;  // Флаг: время синхронизировано по SNTP
}

message DeviceInfo {
  string device_id  = 1;  // Идентификатор устройства
  string fw_version = 2;  // Версия прошивки
}

message ScanBatch {
  DeviceInfo device    = 1;  // Информация об устройстве (опционально)

## 7. Серверная часть (Python TCP-сервер)

### 7.1. Описание

Файл: `python/TCP_server.py`  
Язык: Python 3.11  
Назначение: приём Protobuf-батчей от ESP32, декодирование, вывод в терминал, запись в CSV.

### 7.2. Алгоритм работы

**Инициализация (до начала приёма данных):**

1. Выполняется 10 SNTP-запросов к `pool.ntp.org` (библиотека `ntplib`).
2. Вычисляется среднее смещение между системным временем хоста и NTP-временем (`avg_offset_s`).
3. Создаётся CSV-файл `ble_log_scanBatchProto_csv.csv` с заголовками.

**Приём и разбор пакетов:**

1. TCP-сервер слушает порт `9000` на всех интерфейсах (`0.0.0.0`).
2. Данные накапливаются в буфере `buffer`.
3. Из буфера читаются первые 2 байта — длина payload (`little-endian`).
4. Если в буфере достаточно данных, извлекается payload, выполняется `ScanBatch.ParseFromString(payload)`.
5. Для каждого `BleEvent` в батче вычисляется:
   - `dt_utc_esp` — дата/время из UTC-метки ESP32;
   - `ntp_time_s = sys_time_s - avg_offset_s` — скорректированное NTP-время сервера;
   - `delta = ntp_time_us - utc_esp_ms * 1000` — отклонение меток в мс.
6. Данные выводятся в терминал и дописываются в CSV.

### 7.3. Формат CSV-лога

Файл: `ble_log_scanBatchProto_csv.csv`

| Поле | Описание |
|---|---|
| MAC | MAC-адрес BLE-устройства в формате `XX:XX:XX:XX:XX:XX` |
| RSSI | Уровень сигнала, дБм |
| Monotonic | Монотонное время ESP32, мс |
| UTC (ESP) | Дата/время из метки ESP32 (datetime) |
| UTC Now (SYS) | Системное время сервера в момент приёма |
| UTC Now (NTP) | Скорректированное NTP-время сервера |
| Time sync | Флаг синхронизации времени на ESP32 (True/False) |
| UTC NTP - UTC ESP | Отклонение временны́х меток, мс |

```

### 9.5. Типичные ошибки и их решения

| Ошибка | Причина | Решение |
|---|---|---|
| `ModuleNotFoundError: No module named 'google'` | Protobuf не установлен | `pip install protobuf` |

| `/dev/ttyUSB0` не найден | Программатор не добавлен в USB VirtualBox | Добавить FTDI в настройках USB VirtualBox |
| ESP32 не получает IP | DHCP не настроен или сеть не настроена | Проверить настройки сети адаптера в VirtualBox и ICS на Windows |
| `idf.py: command not found` | Окружение ESP-IDF не активировано | Выполнить `. ~/esp-idf/export.sh` |
| Сервер получает данные, но время не синхронизировано | ESP32 не имеет доступа в Интернет | Проверить ICS на хосте Windows; убедиться, что второй адаптер VirtualBox (NAT) активен |
| Brownout reset при старте | Просадка питания | Добавить конденсатор 300–400 мкФ на 3V3-GND или использовать надёжный БП или отключить Brownout Detector (пункт 5.8) |

- **Синхронизация времени** по SNTP с точностью ±60 мс;
- **Batching** — группировка событий в батчи по количеству (до 10) и по времени (до 100 мс);
- **Сериализация** в формат Protocol Buffers (nanopb) с фреймингом `length-prefix`;
- **Передача по TCP** с автоматическим переподключением при разрыве;
- **Буферизация** офлайн-данных в кольцевом буфере (128 батчей, ~82 КБ ОЗУ) с защитой мьютексом;
- **Сброс буфера** при восстановлении TCP-соединения;
- **Серверная часть** на Python с декодированием Protobuf, коррекцией времени по NTP и записью в CSV;
- **Лабораторный стенд** на VirtualBox/Ubuntu с настройкой сетевого взаимодействия.

### 11.2. Известные ограничения

- **Brownout при старте:** совместная инициализация BLE и Ethernet создаёт пиковую нагрузку на питание. Временное решение (отключение детектора) не является безопасным для продуктивной эксплуатации.
- **Дрейф времени:** ~61 мс/ч обусловлен погрешностью внутреннего RC-генератора ESP32 и часовым интервалом SNTP. Требуется доработка алгоритма коррекции.
- **Нет Wi-Fi сканирования:** в текущей версии реализовано только BLE-сканирование.
- **Нет идентификатора устройства:** поле `DeviceInfo` в ScanBatch не заполняется.
- **Нет TLS:** передача данных ведётся в открытом виде.
- **Нет SPIFFS/NVS логирования:** данные буферизуются только в ОЗУ (теряются при перезагрузке).

### 11.3. Направления дальнейшего развития

- Реализация Wi-Fi сканирования и включение событий Wi-Fi в поток передачи.
- Реализация постоянного локального хранилища (SPIFFS/LittleFS) для накопления данных при длительном отсутствии сети.
- Уменьшение интервала SNTP-синхронизации и реализация плавной коррекции смещения времени для повышения точности UTC-меток до уровня ±10–20 мс.
- Реализация заполнения поля `DeviceInfo` (device_id из MAC, fw_version из константы).
- Добавление TLS-шифрования транспортного уровня.
- Внедрение Watchdog-контроля всех задач FreeRTOS.
- Интеграция с базой данных PostgreSQL на стороне сервера.
- Развёртывание OpenWrt-стенда как промежуточного маршрутизатора в цепи: ESP32 → OpenWrt → сервер БД.
